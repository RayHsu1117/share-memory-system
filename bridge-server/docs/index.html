Bridge Memory MCP Server

讓 claude.ai 與 Claude Code 共用同一份持續累積記憶的橋接服務——部署後完全屬於你自己,資料存在你自己的 Postgres 裡。

點按鈕部署到你自己的 Railway 帳號:只需要填一組 OAUTH_OWNER_PASSWORD(你自己的核准密碼),其餘變數自動設定,等約兩分鐘即可。

步驟一:部署完成後,先拿兩樣東西

  1. 在 Railway 專案裡點 bridge-server 服務 → Settings → Networking,複製你的公開網址(長得像 https://xxx.up.railway.app)。
  2. 同服務的 Variables 頁面,複製 BRIDGE_AUTH_TOKEN 的值(部署時自動產生的隨機密鑰)。

步驟二:產生 Claude Code 連線指令

🔒 這個小工具完全在你的瀏覽器裡執行——只做本地字串組合,沒有任何 fetch、表單送出、分析追蹤或對外連線,你的 token 不會被傳送到任何地方。不放心的話,可以直接檢視本頁原始碼確認(整頁只有這一個檔案,沒有外部腳本)。

步驟三:接上 claude.ai(自訂連接器)

claude.ai → Settings → Connectors → Add custom connector:

  1. 只填 URL:https://xxx.up.railway.app/mcp(不用填任何 Header/token 欄位——claude.ai 的自訂連接器強制走 OAuth,沒有填 Bearer token 的選項)。
  2. claude.ai 會自動探索 OAuth 端點(/.well-known/oauth-protected-resource、/.well-known/oauth-authorization-server)、透過 /oauth/register 動態註冊 client,然後跳出瀏覽器授權頁。
  3. 在授權頁輸入你部署時設定的 OAUTH_OWNER_PASSWORD 核准連線。
  4. 通過後自動導回 claude.ai,連接器顯示已連線,即可使用 save_memory / search_memory 等工具。
⚠️ 已知問題:即使伺服器端 OAuth 完全正確,claude.ai 端仍有已回報的 bug(GitHub anthropics/claude-ai-mcp issue #112、#457),偶爾會在註冊階段失敗,跳出「Couldn't register with sign-in service」。遇到時:
  1. 先用瀏覽器直接打開 https://你的網址/.well-known/oauth-authorization-server,確認有正常回傳 JSON(不是 404 或連不上)。如果這裡就失敗,才是自己伺服器的問題——通常是 PUBLIC_BASE_URL 沒設對或還沒重新部署。
  2. 如果上面那個網址正常,問題大機率在 claude.ai 那端:重新整理再試一次,或稍後再試。這不是你設定錯,是已知的上游限制。
